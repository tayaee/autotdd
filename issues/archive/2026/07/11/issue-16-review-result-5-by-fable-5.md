## 2. 사실확인

### G-1: 유효 타임아웃 +5초 — **사실 (인정)**

커밋된 `autoqafix_core.py:209-248`을 재판독: SIGTERM(즉시) → `proc.wait(1)`
→ `killpg(SIGKILL)` → `proc.wait(2)` → `communicate(2)`. 최악 경로에서
호출자가 지정한 `timeout_sec`보다 **최대 약 5초 늦게** 반환한다. #3의 계산이
맞고, #4는 이 지연 축을 전혀 다루지 않았다(같은 코드에서 R-4 잔존 프로세스만
분석).

### G-2: P2 매트릭스 불완전 — **사실 (인정)**

#4의 해소 현황 표에서 C행은 docstring·wt 재발견·기록 포맷·float 노출만
열거했다. 커밋 코드 재확인 결과 #2의 P2 잔여 항목도 실제로 전부 미해결이다:

- P2#6: `processed = 0  # dispatched + renamed-to-manual` (`autofix.py:371`) — 합산 그대로
- P2#7: `pattern.match` (`autofix.py:109`) — fullmatch 미전환 (단 패턴이 `\.md$`로 끝나 실질 차이는 없음 — 순수 스타일)
- P2#8: `stderr.strip().splitlines()[-3:]` (`autofix.py:324`) — 줄번호 미표시

즉 결론("전부 미해결")은 #4와 #3이 같지만, #4는 **항목별 추적 가능성**에서
#3보다 뒤진다. 다회전 리뷰 사이클에서 매트릭스의 행 단위 완전성은 다음
재작업자(qwen)의 체크리스트 품질로 직결되므로 실질적 부족이다.

### G-3: D-1 독자 분석 부재 — **부분 사실**

#4의 R-1 절은 "prune은 복구가 아니라 메타데이터 청소일 뿐"이라고 prune의
의미론을 이미 짚었고, B-4 행에서 `~/.cache/autoqafix/<cid>` 축적도 지적했다.
그러나 "prune이 **worktree 디렉토리 자체**를 남긴다"는 D-1의 구체 관찰은
교차 검증 절에서 #3을 인용했을 뿐 독자적으로 검증하지 않았다. 절반의 부족.

### G-4: 긍정 변경 미인정 — **사실 (인정, 단 절반은 역방향)**

#4는 A-1·A-4의 해결만 표에 기록했고, qwen이 새로 넣은 D-4(`export
FAKE_MODE="${FAKE_MODE:-ok}"` — 테스트 간 env 누수 차단)를 개선으로 언급하지
않았다. 주니어의 반복 사이클에서 "무엇이 좋아졌는지"의 명시는 수정 방향을
고정시키는 효과가 있으므로 리뷰 태도 차원의 부족이 맞다.

단, D-5는 사정이 다르다 — 3번 항목의 반박 참조(아래 §3).

### G-5: 한계 자인 부재 — **사실 (인정)**

#3은 미재현 항목(A-2 시나리오, SIGTERM 그룹 동작, 프로덕션 PATH)을 말미에
명시했다. #4에는 대응 섹션이 없다. #4에도 실재하는 한계가 있었다:
(a) R-4(SIGTERM 생존 손자)를 **실행으로 재현하지 않고** 코드 추론으로만
결론냈다. (b) R-3의 트리거 조건(SIGKILL 후에도 안 죽는 프로세스)을 실제로
만들지 않았다 — D-state 재현은 비현실적이라 정당하나, 명시했어야 한다.
(c) Windows 경로(taskkill 분기)는 아예 검토 범위 밖이었다.

### G-6: A-1 직접 증거 부재 — **기각 (증거는 있었고, 이제 보강도 완료)**

#4의 A-1 판정 근거는 (i) `autofix.py:344`의 `git add` 코드 판독, (ii) 직접

## 3. 방어 / 변명 / 합리화

**G-1 방어(부분)**: #4의 재작업 체크리스트 5번은 "무조건 그룹 SIGKILL
복원(권장)"이다. 이를 따르면 SIGTERM 유예 시퀀스 자체가 사라져 +5초 지연도
함께 소멸한다 — 즉 처방이 증상을 커버한다. 또한 +5초는 상수 상한이고 운영
타임아웃 10800초 대비 0.05%라 우선순위상 P2 이하다. **변명이 아닌 인정
부분**: "커버된다"는 결과론이고, 분석 자체를 안 한 것은 사실이다. 같은 함수를
읽으면서 정확성 축(R-3·R-4)만 보고 지연 축을 안 본 것은 리뷰 관점의 편향이다.

**G-2 변명**: #4는 신규 유입 결함(R-1·R-2)의 실증에 지면과 시간을 집중했고,
P2 세목은 "C ❌ 전부 미해결"로 뭉뚱그렸다. 우선순위 배분 자체는 옳았다고
본다 — P0 신규 결함 2건은 #3이 놓친 것이고, P2 세목은 #2 문서가 이미
추적하고 있었다. **합리화의 한계**: 표의 행을 3줄 늘리는 비용은 사실상 0이라
"지면 집중"은 온전한 변명이 못 된다. 다음 리뷰부터는 이전 리뷰의 전 항목을
행 단위로 옮긴다.

**G-3 방어**: D-1의 실용적 결과(캐시 축적)는 B-4에서, prune의 의미론적
한계는 R-1에서 이미 다뤘으므로 내용상 공백은 없다. 교차 검증 절에서 #3의
공로로 명시적으로 돌린 것은 중복 서술보다 나은 처리였다.

**G-4 방어(D-5에 한해 역공)**: #3이 칭찬한 D-5(`rm -rf $STATE_DIR/worktree`
stale 정리)는 사실확인 결과 **no-op이다**. 픽스처가 `mktemp -d` 경로라 매 실행
clone_id가 달라지고, 해당 state_dir은 생성 직후라 지울 worktree가 존재한 적이
없다. #4가 이를 "무의미"로 판정한 것이 정확했다. 긍정 변경 인정은 태도로서
바람직하지만, **틀린 칭찬은 주니어에게 잘못된 패턴을 강화**한다 — D-4처럼
실제로 유효한 개선만 인정했어야 하고, #4는 그 D-4마저 누락했으니 절반은
인정, 절반은 #3의 실책이다.

**G-5 인정(방어 없음)**: 한계 자인은 리뷰의 신뢰 보정 장치다. #4는 재현
성공 사례만 나열해 "전부 실증됨"이라는 인상을 줬지만 R-4는 미재현 추론이었다.
이는 방어할 거리가 없는 누락이고, 본 보고서 §2 G-5에 그 한계를 소급 명시했다.

**균형을 위한 기록 — #4가 #3을 바로잡은 3건**: 이 자기 검토가 "부족함"만
남기면 그 또한 왜곡이므로 대조 결과를 함께 적는다. (i) #3 D-3은 reset 삭제를
"A-2 증상 해결"로 긍정 평가했으나 #4의 재현이 origin 오염(P0 R-1)을 증명했다.
(ii) #3은 R-2(FAKE_TARGET 프로덕션 주입)를 통째로 놓쳤다 — B-2 절에서
"dispatch가 매 항목마다 FAKE_TARGET을 덮어쓴다"고 **관찰까지 하고도** 레이어링
위반으로 인식하지 못했다. (iii) #3 D-2의 "보통 None이거나 str"은 부정확 —
`stdout=PIPE`이므로 fallback 값은 항상 `TextIOWrapper` 객체다(R-3).

## 4. 결론

- #4의 부족함은 **사실이다**: 지연 축 미분석(G-1), 매트릭스 불완전(G-2),
  개선 인정 누락(G-4 절반), 한계 자인 부재(G-5). 공통 원인은 "신규 P0 실증에
  집중하면서 이전 리뷰의 전수 추적과 리뷰 자체의 메타 정보(한계·긍정 변경)를
  희생"한 것.
- 그러나 정확성 축에서는 #4가 #3의 오류 3건을 교정했고 P0 2건을 추가
  실증했으므로, 두 리뷰는 **상호 보완**이다. qwen 재작업은 #4의 체크리스트를
  기준으로 하되, #3의 D-1·D-2(지연)·P2 세목·수정 스니펫을 병용할 것.
- 다음 리뷰(#6 예상)의 프로세스 개선: ① 이전 리뷰 전 항목을 행 단위 매트릭스로
  승계, ② 재현 못 한 결론은 "추론"으로 표기하는 한계 절 필수, ③ 유효한 긍정
  변경 명시(단 유효성 검증 후), ④ 저장 전 기존 번호 `ls` 확인.
