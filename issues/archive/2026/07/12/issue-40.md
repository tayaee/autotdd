# issue-40: autotddreviewfix 리뷰어 프롬프트 4부 구조 — 증거 계약으로 추측 리뷰 차단
agent-tier: any

## 배경

issue-20 감사에서 리뷰 지적 40건 중 27.5%가 오판이었고 qwen은 55%였다.
오판의 지배적 패턴은 "근거 인용 없이 일반론으로 단정" — 추측 리뷰다.
현행 autotddreviewfix Step 2의 리뷰어 프롬프트는 한 줄("issue-N의 구현에
대해 코드 품질 감사를 수행하여 … 파일을 작성해")뿐이라, 무엇을 어떤
근거 수준으로 보고해야 하는지가 전적으로 리뷰어 재량이다.

grill 세션(2026-07-12) 합의: 프롬프트를 4부 구조로 재작성해 추측을
구조적으로 차단한다. 철학은 정밀도 우선 — 리뷰 사이클은 수백 티켓 동안
반복되므로 한 번에 다 찾을 필요 없고, 확실한 것만 잡는 게 중요하다.
누락은 다음 사이클이 회수하지만 오판은 무인 `/autotdd` 풀사이클 비용을
낭비시킨다.

**선행**: issue-39 (리뷰 파일명 `issue-N__TYPE-code-review__BY-<base>.md`).

## 요구사항

autotddreviewfix SKILL.md Step 2의 리뷰어 프롬프트(외부 래퍼·셀프 리뷰
서브에이전트 공통)를 다음 4부 구조로 재작성한다:

1. **① 환경 사실**:
   - Python 버전은 대상 리포의 `.python-version`에서, 없으면
     `pyproject.toml`(`requires-python`)에서 읽어 프롬프트에 명시하도록
     지시한다. 판정 규칙: 버전이 3.12 미만이면 그 버전 준수를 요구하고,
     3.12 초과면 언어 준수 수준을 3.12로 캡한다.
   - 코드는 이미 ruff+pyright+pytest+회귀를 통과한 상태임을 알린다.
2. **② 리뷰 범위**: 기계가 못 잡는 것만 — 로직 오류, 스펙(이슈 요구사항)
   불일치, 보안(OWASP Top 10 렌즈), 동시성·경계조건. **ruff/pyright가
   잡는 스타일·타입 지적은 보고 금지** (파이프라인의 다른 단계 몫).
3. **③ 증거 계약**: finding마다 필수 3요소 —
   - `파일:라인` + 실제 코드 인용
   - 문제가 재현되는 구체 실패 시나리오 (입력/상태 → 잘못된 결과)
   - 확인 방법 (실행·재현 절차)
   그리고 명시: "확인하지 못한 것은 쓰지 마라. 리뷰 사이클은 반복되므로
   이번에 다 찾을 필요 없다 — 누락보다 오판이 비싸다."
4. **④ 구조화 finding 포맷**: 자유 산문 금지. finding당 고정 필드
   (파일:라인 / 코드 인용 / 실패 시나리오 / 확인 방법 / 심각도 제안
   must-fix·good-to-fix)를 표 또는 고정 섹션 형식으로 강제 — 플래너
   (issue-41)의 형식 게이트가 기계적으로 판정할 수 있게.
5. 기존 지시("본문 첫 줄에 자기 모델명(버전 포함) 기입")는 유지.
6. 셀프 리뷰 서브에이전트 프롬프트에도 동일 4부 구조 적용.
