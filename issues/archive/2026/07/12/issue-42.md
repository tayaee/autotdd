# issue-42: autotddreviewfix 리뷰 범위 고정 — 리뷰어에게 diff·파일 목록 제공
agent-tier: any

## 배경

내장 /code-review ultra와의 비교(grill 세션 2026-07-12)에서 차용이 확정된
항목. ultra는 리뷰 범위가 브랜치 diff로 명확히 고정되는 반면, 현행
autotddreviewfix Step 2는 리뷰어에게 "issue-N의 구현"이라고만 알려준다 —
리뷰어가 무엇이 변경됐는지부터 추측해야 하므로, 추측 리뷰를 구조적으로
유발한다(리포 전체를 뒤지다 무관한 코드를 지적하는 패턴). 리뷰 대상을
명시하는 것은 issue-40 증거 계약의 전제조건이다.

**선행**: issue-39 (파일명 규약 v2). issue-40과 병행 가능.

## 요구사항

1. **구현 범위 캡처**: Step 1(Coder MVP) 시작 직전 실행 세션이 `HEAD`
   해시를 기록하고, 완료 직후 범위를 산출한다:
   - 커밋 범위: `<시작HEAD>..HEAD`
   - 변경 파일 목록: `git diff --name-only <시작HEAD>..HEAD`
   - Step 1을 skip한 재실행(멱등 재개)의 경우: 이슈 번호가 커밋 메시지에
     포함되는 기존 관행(`issue-N:` prefix)으로 해당 커밋들을 역추적한다.
   - worktree 모드: 병합 후 main 기준으로 동일하게 산출.
2. **Step 2 프롬프트에 범위 주입**: 리뷰어 프롬프트에 다음을 포함한다 —
   - 이슈 파일 경로 (`issues/issue-N*.md`) — 스펙 대조용
   - 커밋 범위와 변경 파일 목록
   - 회귀 스크립트 경로 (`regression-tests/verify-issue-N.sh`)
   - 지시: "리뷰 대상은 이 범위다. 범위 밖 코드는 이 변경과의 직접
     상호작용(호출부·피호출부)이 문제일 때만 언급하라."
3. **셀프 리뷰 경로에도 동일 적용**.
4. **범위 산출 실패 시**: 커밋을 역추적하지 못하면 리뷰어에게 범위 없이
   맡기지 말고, 이슈 파일과 회귀 스크립트 경로만이라도 명시하고 그 사실을
   프롬프트에 밝힌다 (침묵 금지).
